Specification — §1

Everything production
storage actually needs.

Not a MinIO wrapper. Not a proof-of-concept. Every feature on this list ships in the 0.1.0 binary with test coverage and published behavior.

  1. 01
    S3-compatible API with real SigV4

    Mount an aws-sdk client at /s3/* and it just works. Authorization headers are verified, not parsed-and-trusted — the canonical request is rebuilt from what the server received, HMACs compared in constant time. Tampered URLs are rejected with 403.

  2. 02
    Server-side encryption, dedup-friendly

    AES-256-GCM at rest. Per-blob keys derived via HKDF from a master key and the plaintext SHA-256, so two identical uploads produce identical ciphertext — content-addressable deduplication keeps working.

  3. 03
    Versioning, delete-markers, object lock

    Full version history per key, with ?version_id= on GET/DELETE and a one-shot purge endpoint. Per-object retention (WORM) and legal hold block deletes with HTTP 409 until the clock runs out.

  4. 04
    Async primary-replica replication

    Every write appends an atomic event to a log; replicas pull the log, fetch content-addressed blobs, and apply directly to their own database. One CLI command promotes a replica when the primary fails.

  5. 05
    Image transforms, on the fly

    Resize / crop / convert via URL query parameters. Auto-negotiates AVIF → WebP → JPEG from the Accept header. Results go through a two-level cache (memory LRU + bounded disk) so your CDN actually helps.

  6. 06
    Lifecycle, quotas, audit log

    Expire-by-prefix rules run on schedule or on demand. Bucket-level quotas (size + object count). Every write is logged to SQLite with actor, action, target, and source IP.

  7. 07
    Prometheus, Grafana, and a dashboard

    Metrics at /metrics, an embedded admin UI at /dashboard, and a ready-to-import Grafana JSON that covers p50/p95/p99, error ratio, and throughput — no dashboard-building afternoon required.

  8. 08
    Official SDKs

    TypeScript (zero-dep, native fetch), Python (httpx + typed dataclasses), and Go (zero-dep net/http). Full coverage of versioning, object lock, and lifecycle — not just CRUD.

— code · §2 
# Create a bucket, upload an image, resize it on the fly.

curl -X POST http://localhost:8000/v1/buckets \
  -H "Authorization: Bearer $VFS_KEY" \
  -d '{"name": "photos"}'

curl -X PUT http://localhost:8000/v1/objects/photos/beach.jpg \
  -H "Authorization: Bearer $VFS_KEY" \
  -H "Content-Type: image/jpeg" \
  --data-binary @beach.jpg

# WebP at 400px, with cache on second hit.
curl "http://localhost:8000/v1/objects/photos/beach.jpg?w=400&format=webp" \
  -H "Authorization: Bearer $VFS_KEY" -o beach.webp
import { VaultFS } from "@vaultfs/client";

const vfs = new VaultFS({
  baseUrl: "http://localhost:8000",
  apiKey:  process.env.VFS_KEY!,
});

await vfs.enableVersioning("photos");
await vfs.putObject("photos", "beach.jpg", file, "image/jpeg");

// Lock the file for 30 days with a legal hold.
await vfs.setLock("photos", "beach.jpg", {
  retain_until: new Date(Date.now() + 30 * 86_400_000).toISOString(),
  legal_hold:   true,
});
from vaultfs import VaultFS

vfs = VaultFS("http://localhost:8000", api_key=os.environ["VFS_KEY"])

vfs.enable_versioning("photos")
vfs.put_object("photos", "beach.jpg",
               open("beach.jpg", "rb"), "image/jpeg")

# Every PUT creates a new version on a versioned bucket.
versions = vfs.list_versions("photos", "beach.jpg")
for v in versions:
    print(v.version_id, v.size, v.is_latest)
import "github.com/aniasse/vaultfs/sdks/go"

c := vaultfs.New("http://localhost:8000", os.Getenv("VFS_KEY"))

// Seven-day expiry under tmp/.
rule, _ := c.CreateLifecycleRule("photos", "tmp/", 7)
fmt.Println("rule id:", rule.ID)

// Sweep now; don't wait for the schedule.
expired, freed, _ := c.RunLifecycle()
fmt.Printf("expired %d objects, freed %d bytes\n", expired, freed)
# Replicate a primary onto this machine, every 5 seconds.
vaultfsctl replicate \
  --url          http://localhost:8000 \
  --key          $REPLICA_KEY \
  --primary      https://primary.example.com \
  --primary-key  $PRIMARY_KEY \
  --cursor-file  /var/lib/vaultfs/replica.cursor \
  --interval     5

# Primary went down? Promote the replica.
vaultfsctl promote \
  --url          http://localhost:8000 \
  --key          $REPLICA_KEY \
  --cursor-file  /var/lib/vaultfs/replica.cursor
Architecture — §3

Boring, on purpose.

One process. One data directory. A SQLite file for metadata, content-addressed blobs in blobs/<aa>/<bb>/<sha>, and an optional append-only replication log. Backup is rsync. Restore is rsync.

— get started · §4

Pick a door.
Walk in.

Three ways to start. All of them produce the same running daemon — pick whichever your fingers already know. On first launch the server generates an admin API key and prints it to stdout. Save it.

Option A

Docker

Zero configuration. One volume for state. Production-ready image on GHCR.

docker run -d \
  --name vaultfs \
  -p 8000:8000 \
  -v vaultfs-data:/data \
  ghcr.io/aniasse/vaultfs:latest
Option B

Binary

Linux amd64 or macOS arm64. Checksums in SHA256SUMS on the release page.

# linux-amd64
curl -LO https://github.com/aniasse/vaultfs/\
releases/download/v0.1.0/\
vaultfs-linux-amd64.tar.gz
tar xzf vaultfs-linux-amd64.tar.gz
./vaultfs
Option C

From source

Rust 1.85+ (edition2024). Full test suite runs in ~2 s after a warm build.

git clone \
  https://github.com/aniasse/vaultfs.git
cd vaultfs
cargo build --release
./target/release/vaultfs
