fonctionnalités

Tout ce dont un stockage
de production a vraiment besoin.

Ni un wrapper MinIO, ni un POC. Chaque élément de cette liste figure dans le binaire 0.1.0 avec sa couverture de tests et son comportement documenté.

  1. 01
    API S3 avec SigV4 vérifiée

    Branchez un client aws-sdk sur /s3/* et ça fonctionne. Les en-têtes Authorization sont vérifiés, pas simplement parsés — la requête canonique est reconstruite depuis ce que le serveur a reçu, les HMAC comparés en temps constant. Une URL altérée renvoie 403.

  2. 02
    Chiffrement au repos, compatible dédup

    AES-256-GCM au repos. Clés par blob dérivées via HKDF d'une clé maître et du SHA-256 du plaintext : deux uploads identiques produisent le même ciphertext, la dédup content-addressable continue de fonctionner.

  3. 03
    Versioning, delete-markers, object lock

    Historique complet par clé, avec ?version_id= sur GET/DELETE et un endpoint de purge en une seule requête. La rétention par objet (WORM) et le legal hold bloquent les suppressions avec un HTTP 409 jusqu'à ce que la rétention expire.

  4. 04
    Réplication primary-replica asynchrone

    Chaque écriture ajoute un événement atomique à un log ; les répliques le consomment, récupèrent les blobs content-addressed, et appliquent directement dans leur base. Une commande CLI promeut une réplique quand le primary tombe en panne.

  5. 05
    Transformations d'images, à la volée

    Resize / crop / conversion via paramètres d'URL. Négociation automatique AVIF → WebP → JPEG via l'en-tête Accept. Résultats passés dans un cache à deux niveaux (LRU mémoire + disque borné), pour que votre CDN ait un vrai intérêt.

  6. 06
    Lifecycle, quotas, audit log

    Règles d'expiration par préfixe, à la demande ou programmées. Quotas par bucket (taille + nombre d'objets). Chaque écriture est enregistrée en SQLite avec acteur, action, cible et IP source.

  7. 07
    Prometheus, Grafana, et un dashboard

    Métriques sur /metrics, UI admin embarquée sur /dashboard, et un JSON Grafana prêt à importer qui couvre p50/p95/p99, taux d'erreur et débit — plus besoin de passer une après-midi à construire un dashboard.

  8. 08
    SDKs officiels

    TypeScript (zéro dépendance, fetch natif), Python (httpx + dataclasses typées), et Go (zéro dépendance, net/http standard). Couverture complète du versioning, de l'object lock et du lifecycle — pas juste du CRUD.

Parlez-lui.

code 
# Créer un bucket, uploader une image, la redimensionner à la volée.

curl -X POST http://localhost:8000/v1/buckets \
  -H "Authorization: Bearer $VFS_KEY" \
  -d '{"name": "photos"}'

curl -X PUT http://localhost:8000/v1/objects/photos/beach.jpg \
  -H "Authorization: Bearer $VFS_KEY" \
  -H "Content-Type: image/jpeg" \
  --data-binary @beach.jpg

# WebP à 400px, avec cache dès la seconde requête.
curl "http://localhost:8000/v1/objects/photos/beach.jpg?w=400&format=webp" \
  -H "Authorization: Bearer $VFS_KEY" -o beach.webp
import { VaultFS } from "@vaultfs/client";

const vfs = new VaultFS({
  baseUrl: "http://localhost:8000",
  apiKey:  process.env.VFS_KEY!,
});

await vfs.enableVersioning("photos");
await vfs.putObject("photos", "beach.jpg", file, "image/jpeg");

// Verrouille le fichier 30 jours avec un legal hold.
await vfs.setLock("photos", "beach.jpg", {
  retain_until: new Date(Date.now() + 30 * 86_400_000).toISOString(),
  legal_hold:   true,
});
from vaultfs import VaultFS

vfs = VaultFS("http://localhost:8000", api_key=os.environ["VFS_KEY"])

vfs.enable_versioning("photos")
vfs.put_object("photos", "beach.jpg",
               open("beach.jpg", "rb"), "image/jpeg")

# Chaque PUT crée une nouvelle version sur un bucket versionné.
versions = vfs.list_versions("photos", "beach.jpg")
for v in versions:
    print(v.version_id, v.size, v.is_latest)
import "github.com/aniasse/vaultfs/sdks/go"

c := vaultfs.New("http://localhost:8000", os.Getenv("VFS_KEY"))

// Expiration à 7 jours sous tmp/.
rule, _ := c.CreateLifecycleRule("photos", "tmp/", 7)
fmt.Println("rule id:", rule.ID)

// Balaye maintenant, sans attendre la prochaine exécution.
expired, freed, _ := c.RunLifecycle()
fmt.Printf("expired %d objects, freed %d bytes\n", expired, freed)
# Répliquer un primary sur cette machine, toutes les 5 secondes.
vaultfsctl replicate \
  --url          http://localhost:8000 \
  --key          $REPLICA_KEY \
  --primary      https://primary.example.com \
  --primary-key  $PRIMARY_KEY \
  --cursor-file  /var/lib/vaultfs/replica.cursor \
  --interval     5

# Primary tombé ? On promeut la réplique.
vaultfsctl promote \
  --url          http://localhost:8000 \
  --key          $REPLICA_KEY \
  --cursor-file  /var/lib/vaultfs/replica.cursor
architecture

Ennuyeux. C'est le but.

Un processus. Un dossier de données. Un fichier SQLite pour les métadonnées, des blobs content-addressed dans blobs/<aa>/<bb>/<sha>, et un log de réplication append-only optionnel. Sauvegarde : rsync. Restauration : rsync.

  curl / SDK   ───── HTTP ─────▶   ┌────────────────────┐         ┌───────────┐
  aws-sdk-s3   ──── SigV4 ─────▶   │  axum + tokio      │         │           │
  dashboard    ────  TLS  ─────▶   │  ────────────────  │         │  nœud     │

démarrer

Choisissez une porte.
Entrez.

Trois façons de démarrer. Toutes produisent le même daemon — prenez celle que vos doigts connaissent déjà. Au premier lancement, le serveur génère une clé API admin et l'affiche sur stdout. Gardez-la.

Option A · Docker

Docker

Zéro configuration. Un volume pour les données. Image prête pour la prod sur GHCR.

docker run -d \
  --name vaultfs \
  -p 8000:8000 \
  -v vaultfs-data:/data \
  ghcr.io/aniasse/vaultfs:latest
Option B · Binaire

Binaire

Linux amd64 ou macOS arm64. Checksums dans SHA256SUMS sur la page de release.

# linux-amd64
curl -LO https://github.com/aniasse/vaultfs/\
releases/download/v0.1.0/\
vaultfs-linux-amd64.tar.gz
tar xzf vaultfs-linux-amd64.tar.gz
./vaultfs
Option C · Sources

Depuis les sources

Rust 1.85+ (edition2024). La suite de tests complète tourne en ~2 s après un build à chaud.

git clone \
  https://github.com/aniasse/vaultfs.git
cd vaultfs
cargo build --release
./target/release/vaultfs

$ whoami
Possédez le disque, les clés, et l'uptime. Spec API complète, runbooks et benchmarks dans le dépôt.
